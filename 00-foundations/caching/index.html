Caching Strategies

Caching improves system performance and scalability by storing frequently accessed data in memory, reducing expensive database and disk operations.

What is Caching?

Caching stores frequently accessed data in memory so applications can retrieve it quickly without repeatedly hitting databases or disks.

Memory access is significantly faster than disk access or network calls.

Basic Cached Architecture

Client
   ↓
Load Balancer
   ↓
Application Servers
   ↓
Caching Layer (Redis/Memcached)
   ↓
Database

Instead of every request directly hitting the database, the application first checks the cache for data.

Cache Hit vs Cache Miss

Type Description
Cache Hit Requested data exists in cache and is returned quickly
Cache Miss Requested data is not in cache and must be fetched from database

Read Heavy Systems Benefit Most

Caching works best when systems perform significantly more reads than writes. 

Wikipedia Example

Reads: Millions ✅
Writes: Few ❌

Perfect caching use case

Common Cache Problems

Problem Description Solution
Stale Data Cache contains outdated information Use expiration policies or invalidation
Hotspot / Celebrity Problem One key receives massive traffic Replicate hot data across multiple cache nodes
Cold Start Problem Empty cache overloads database after restart Warm up cache before serving traffic

Eviction Policies

Since cache memory is limited, eviction policies decide what data gets removed when memory becomes full.

Policy Description
LRU Least Recently Used item removed first
LFU Least Frequently Used item removed first
FIFO First inserted item removed first

Why Caching Matters in System Design

Large-scale systems cannot allow every request to directly hit databases because disk access and network calls are expensive. 

Without Cache:
Application → Database → Disk

With Cache:
Application → Memory Cache ✅

Scalability Benefits

  • Reduces database load
  • Improves response latency
  • Increases throughput
  • Allows independent cache scaling
  • Reduces cross-data-center traffic

Distributed Cache Architecture

Cache systems are often horizontally distributed across multiple cache servers using hashing mechanisms. 

Application Server
      ↓
Hash(Key)
      ↓
Specific Cache Node

Cache Expiration Strategy

Expiration policies determine how long data remains cached before becoming stale.

Long expiration → Better performance but stale data risk

Short expiration → Fresh data but lower cache effectiveness

Architectural Tradeoffs

Benefit Tradeoff
Lower latency Potential stale data
Higher throughput Cache invalidation complexity
Reduced DB load Additional infrastructure

Hotspot / Celebrity Problem

Some keys become extremely popular and overload a single cache node. 

Example:
Millions search for "Brad Pitt"

One cache node overloaded ❌
Solution: Replicate hot data across multiple cache servers.

Cold Start Problem

If cache servers restart empty, all traffic suddenly hits the database. 

Empty Cache ❌
↓
Massive DB Traffic ❌
↓
Database Overload ❌
Solution: Pre-warm caches using historical traffic or background loading.

LRU Cache Internal Design

LRU caches commonly combine:

  • HashMap → O(1) lookup
  • Doubly Linked List → maintain usage order
Most Recently Used ← Head

Least Recently Used ← Tail

Q: Why is caching important?

Caching improves performance and scalability by reducing expensive database and disk operations.

Q: What is a cache hit and cache miss?

Cache hit means requested data exists in cache. Cache miss means data must be fetched from database.

Q: What is the hotspot problem?

Extremely popular keys overload specific cache nodes due to uneven traffic distribution.

Q: What is the cold start problem?

Empty caches after restart can overload databases because every request becomes a cache miss.

Q: Explain LRU cache.

LRU removes the least recently used item first using a HashMap and Doubly Linked List.

Q: Redis vs Memcached?

Redis supports advanced features like persistence, pub/sub, and transactions while Memcached is simpler and lightweight.
