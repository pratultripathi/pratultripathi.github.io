CAP Theorem

CAP ┌──────────────────────────┐ │ C → Consistency │ │ A → Availability │ │ P → Partition Tolerance │ └───────────────────────── ┘
CAP theorem explains the tradeoffs in distributed databases.
Property Meaning
Consistency Every read gets latest write
Availability System always responds
Partition Tolerance System survives network failures
When network partition happens: Choose ONE: Consistency OR Availability
Memory Trick → CAP = Distributed System Tradeoffs

CAP Categories

Type Focus Examples
CP Consistency + Partition Tolerance MongoDB, Google Spanner
AP Availability + Partition Tolerance Cassandra, DynamoDB
CA Consistency + Availability MySQL, Oracle

Eventual Consistency

Data becomes consistent over time.
Update profile picture US Server → New image ✅ EU Server → Old image temporarily ❌ Eventually: All servers sync ✅

Interview Question

Q: What is eventual consistency?

A: All replicas eventually converge to the same value, but temporary stale reads are allowed.
